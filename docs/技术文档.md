# 技术文档

## 项目概览

`shoe-factory-assistant` 当前阶段实现订单 Excel 上传、订单基础信息识别、订单/装箱单 PDF 预览打印，以及订单明细、装箱单明细的手动识别。

当前上传的 Excel 通常包含 `订单`、`装箱单`、`外箱贴标`、`包装要求` 等 sheet。上传后先保存原始文件并读取基础信息，订单单和装箱单可以直接基于原 Excel sheet 转 PDF 打印。外箱贴标、内盒贴标、出货单等后续打印依赖完整识别后的明细数据。

## 模块划分

- `backend`: 负责原稿保存、基础信息识别、订单/装箱单识别、订单查询、PDF 生成和 PDF 文件读取。
- `web`: 负责打印列表、订单列表、订单详情、订单上传、打印类型选择和 PDF 预览。
- `print-agent`: 第二阶段负责连接本地打印设备并执行真实打印。
- `sql`: 负责数据库初始化和表结构脚本。

## 技术约定

- 后端使用 Java 17、Spring Boot 3、MyBatis-Plus、MySQL 8。
- Excel 读取和打印设置使用 Apache POI。
- Excel 转 PDF 优先通过 LibreOffice 命令行完成，Windows 上可用 Microsoft Excel COM 兜底。
- 前端使用 React、Vite、TypeScript、Ant Design。

## 核心数据流

1. 上传 Excel 原稿，文件保存到 `D:/清化资料/年份/客户/月`。
2. 上传阶段只读取基础信息，写入 `order_record`，包括订单流水号、客户、开发编号汇总、总双数和总箱数。
3. 上传后创建订单、装箱单两条 `order_sheet_print_task`，原稿文件名和本地路径写入任务表。
4. 打印订单或装箱单时，系统从对应打印任务读取原 Excel 的 `订单` 或 `装箱单` sheet，调整打印格式后转成 PDF。
5. 用户手动触发识别订单后，系统从订单打印任务读取原稿并解析 `订单` sheet，写入 `order_record_detail`。
6. 用户手动触发识别装箱单后，系统从装箱单打印任务读取原稿并解析 `装箱单` sheet，写入 `order_packing_detail`。
7. `order_detail_process` 保存订单明细对应的订包装、定大底、定中底等处理状态。
8. `order_sheet_print_task` 保存订单/装箱单两个整单级 sheet 的原稿路径、打印状态、PDF 路径、打印次数。

订包装中“一行装箱明细代表一个箱子”以及 `14装～2装` 的汇总口径，统一以 [`订包装业务规则.md`](订包装业务规则.md) 为准。

## 打印任务设计

订单单和装箱单属于整单级打印，跟 `order_record.id` 走。上传一个 Excel 后会自动创建两条 `order_sheet_print_task`：

- `print_type = 1`：订单 sheet。
- `print_type = 2`：装箱单 sheet。

原稿文件名和路径写入 `original_file_name`、`original_file_path`。PDF 预览成功后写入 `preview_pdf_path` 和 `pdf_generated_at`。生成失败时任务状态写为 `FAILED`，失败原因写入 `error_message`；重新生成成功后会清空失败原因，并按是否已有打印次数恢复为 `PENDING` 或 `PRINTED`。

外箱贴标、内盒贴标、标签、出货单等属于明细级打印，跟 `order_record_detail.id` 走，后续单独建表，不放进 `order_sheet_print_task`。

详细设计见：[打印任务与识别流程设计](./打印任务与识别流程设计.md)。
